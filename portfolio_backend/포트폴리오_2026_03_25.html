03 ﻿자기 보정 타이머를 통한 누적 오차 해결
문제

﻿setTimeout을 재귀적으로 연결하여 페이즈(Morning → Evening → Night)를 전환할 때, 설정한 시간보다 페이즈가 늦게 시작되는 현상 발생.

원인

﻿자바스크립트의 이벤트 루프 지연(Event Loop Latency). 호출 스택이 비어잇지 않으면 setTimeout에 지정된 시간이 지나도 콜백이 즉시 실행되지 않으므로, 매 실행마다 수 밀리초(ms)의 오차가 발생하고 이것이 시간 누적 오차(Accumulated Error)로 이어짐

해결

﻿단순 지연 시간(delay) 설정 방식에서 벗어나, 드리프트 보상(Drift Compensation) 알고리즘을 적용
- 기대 실행 시점(Expected Time) : [초기 시작 시점 + 누적된 페이즈 시간의 합]을 통해 이론상 실행되어야 할 절대적인 시점을 산출
- 오차(Drift) 계산: [실제 실행 시점 - 기대 실행 시점]을 통해 발생한 지연을 밀리초(ms) 단위로 파악
- 보상(Compensation) : 산출된 오차를 다음 페이즈의 실행 간격에서 차감하여, 시스템 전체 타임라인이 특정 시점에 수렴하도록 설정

드리프트 오차 = [초기값 + 다음 페이즈 절대값] - [실제 시간]
04 JMeter 부하 테스트 — 서버 가용 리소스 산출
문제

동시 접속자 수가 증가함에 따라 서버가 요청을 처리하지 못하고 마비되는 현상 발생.

JMeter 테스트 결과 : 특정 부하 이상에서 응답 시간이 최대 **21,000ms(21초)**까지 치솟으며 사실상 서비스 이용이 불가능한 가용성 상실 상태 확인.

원인

서버 마비 시점의 Docker 리소스 점유율 분석 결과, Memory는 60% 미만으로 여유가 있었으나 CPU 사용률이 할당량(10%)을 100% 점유하는 것을 포착.

분석 결과, 메모리 부족이 아닌 CPU 연산 자원의 한계가 전체 시스템의 성능 저하를 야기하는 핵심 원인임을 데이터로 입증.

해결

성능 최적화 : CPU 자원 재배분 및 분산 처리를 통해 서버 마비 상태를 해소하고, 응답 속도를 약 52% 단축.

확장성 검증 : CPU 10% 서버 2대 분산 환경이 고사양 서버 1대와 대등한 성능을 보임을 확인하여, 향후 트래픽 증가에 유연하게 대응할 수 있는 수평 확장(Scale-out) 구조의 기반 확보.

로비 서버 — CPU 10% / 128MB
Loop Load Test — 응답시간 21,000ms
로비 서버 — CPU 20% / 128MB
Loop Load Test — 응답시간 11,000ms (↓50%)
